# P1 数据库基础讲解

本文解释数据库连接、数据库健康检查、测试替身和 Alembic 迁移基线。

## 1. 请求流程

```text
GET /api/v1/health/database
-> api/v1/routes/health.py
-> Depends(get_db)
-> core/database.py 创建 Session
-> services/health.py 执行 SELECT 1
-> schemas/health.py 校验成功响应
-> 返回 HTTP 200 或 HTTP 503
-> get_db 关闭 Session
```

## 2. core/database.py

```python
engine = create_engine(settings.DATABASE_URL, pool_pre_ping=True)
```

- `settings.DATABASE_URL` 是数据库地址。
- `engine` 管理 SQLAlchemy 与数据库之间的连接能力。
- 创建 Engine 时通常不会立即连接数据库。
- `pool_pre_ping=True` 会在复用连接前检查连接是否可用。

```python
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
```

- `sessionmaker` 是 Session 工厂。
- 每个请求应该获得自己的 Session。
- `autocommit=False` 表示业务代码明确决定何时提交事务。
- `autoflush=False` 表示不会在查询前自动把待处理修改写入数据库。
- `bind=engine` 把 Session 与当前数据库 Engine 关联。

```python
Base = declarative_base()
```

- `Base` 是 ORM Model 的共同父类。
- 后续的 `User`、`Song` 等表模型都会继承它。
- Alembic 通过 `Base.metadata` 读取所有已注册的数据表结构。

```python
def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

- `SessionLocal()` 创建一次请求使用的数据库会话。
- `yield db` 把会话交给 FastAPI 接口。
- 请求处理结束后进入 `finally`。
- `db.close()` 释放 Session 占用的连接资源。

## 3. services/health.py

```python
db.execute(text("SELECT 1"))
```

- `SELECT 1` 不读取业务表，只验证数据库能否执行最简单的 SQL。
- `text()` 把 SQL 字符串包装成 SQLAlchemy 可执行对象。

```python
except SQLAlchemyError:
    return False
```

- 数据库连接失败、超时或 SQL 执行失败通常会转换为 `SQLAlchemyError` 子类。
- Service 不把底层异常详情直接返回给外部用户，避免泄露内部信息。
- Route 根据布尔结果决定 HTTP 状态码。

## 4. routes/health.py

```python
def database_health_check(db: Session = Depends(get_db)):
```

- `Depends(get_db)` 是 FastAPI 依赖注入。
- Route 不需要自己创建和关闭 Session。
- 类型 `Session` 帮助编辑器和开发者理解 `db` 的能力。

```python
raise HTTPException(
    status_code=status.HTTP_503_SERVICE_UNAVAILABLE,
    detail="database connection is unavailable",
)
```

- HTTP 503 表示服务当前依赖不可用。
- 它不同于 500：503 明确说明服务暂时不能正常提供能力。

## 5. schemas/health.py

`DatabaseHealthResponse` 定义数据库可用时的响应结构：

```json
{
  "status": "healthy",
  "database": "postgresql",
  "detail": "database connection is available"
}
```

如果 Route 返回了缺失字段或错误类型，Pydantic 会阻止不符合契约的响应。

## 6. tests/test_health.py

测试不连接真实 PostgreSQL，而是使用两个测试替身：

- `WorkingSession`：执行查询时不报错，模拟数据库可用。
- `FailingSession`：抛出 `SQLAlchemyError`，模拟数据库不可用。

```python
app.dependency_overrides[get_db] = override_get_db
```

这行把正式的 `get_db` 临时替换为测试版本。这样测试具有以下特点：

- 不依赖本机是否安装 PostgreSQL。
- 执行速度快。
- 可以稳定复现成功和失败场景。
- 不会修改真实数据。

测试结束后必须执行：

```python
app.dependency_overrides.clear()
```

否则一个测试的替换规则可能污染后续测试。

## 7. Alembic

Alembic 用版本文件管理数据库结构变化。

核心文件：

```text
backend/alembic.ini
backend/alembic/env.py
backend/alembic/versions/
```

`env.py` 中的关键关系：

```python
target_metadata = Base.metadata
```

它告诉 Alembic：以后比较数据库结构时，以项目 ORM Model 的 metadata 为代码侧依据。

```python
config.set_main_option("sqlalchemy.url", settings.DATABASE_URL.replace("%", "%%"))
```

它让 Alembic 与应用读取同一个数据库环境变量，避免维护两份连接地址。

当前首个迁移是空迁移，因为项目还没有业务数据表。它建立了迁移历史起点。
