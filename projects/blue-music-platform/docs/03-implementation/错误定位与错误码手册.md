# 错误定位与错误码手册

本文用于快速定位蓝乐平台在本地开发、Docker 和云服务器运行时出现的错误。

适用范围：

- FastAPI 接口错误。
- PostgreSQL 和 Redis 连接错误。
- 登录、Token 和权限错误。
- 后续 Agent 工作流和外部平台错误。
- Docker 与云服务器部署错误。

## 1. 一分钟定位法

遇到错误时，按下面顺序处理：

```text
1. 记录自己刚才执行了什么操作
2. 查看 HTTP 状态码
3. 查看 error.code
4. 复制 request_id；后台任务同时复制 task_id
5. 在日志中搜索编号
6. 根据本手册中的错误码和检查命令定位
```

接口错误示例：

```json
{
  "error": {
    "code": "DATABASE_UNAVAILABLE",
    "message": "数据库连接暂时不可用",
    "request_id": "3f564950a27c45c39b00ec853c5979cd"
  }
}
```

这里最重要的是：

```text
HTTP状态码：503
错误码：DATABASE_UNAVAILABLE
请求编号：3f564950a27c45c39b00ec853c5979cd
```

## 2. request_id 和 task_id

### 2.1 request_id

`request_id` 是一次 HTTP 请求的唯一编号，可以理解为本次操作的小票编号。

```text
浏览器发送请求
-> 后端生成 request_id
-> 后端响应头返回 X-Request-ID
-> 错误 JSON 返回同一个 request_id
-> 后端日志记录同一个 request_id
```

响应头与错误 JSON 中的编号必须一致。

如果 Nginx 等可信代理传入格式正确的 `X-Request-ID`，后端会继续使用该编号；没有传入或格式不正确时，后端生成新的 32 位编号。

`request_id` 只用于追踪问题，不代表身份，也不能代替 Token。

### 2.2 task_id

`task_id` 用于追踪后续耗时较长的 Agent 任务。

```text
用户点击运行
-> HTTP请求获得request_id
-> 系统创建任务并生成task_id
-> 爬取
-> 分析
-> 作词
-> 音乐创作
```

同一个后台任务的每个步骤都要记录相同的 `task_id`。本阶段已经为日志预留 `task_id` 字段，任务系统开发时再接入。

## 3. 错误响应字段

统一结构：

```json
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "请求参数校验失败",
    "request_id": "30ae049c458d4cf6b35a73a3ade43363",
    "detail": [
      {
        "location": "body.username",
        "message": "Field required",
        "type": "missing"
      }
    ]
  }
}
```

字段说明：

| 字段 | 用途 | 是否稳定 |
|---|---|---|
| `code` | 程序和手册使用的错误码 | 稳定，不随意修改 |
| `message` | 给用户看的中文说明 | 可以优化措辞 |
| `request_id` | 在日志中定位本次请求 | 每次请求不同 |
| `detail` | 安全、结构化的字段错误 | 仅在必要时出现 |

`detail` 不返回原始输入值，避免把密码、Token 或其他敏感内容带回响应。

## 4. HTTP 状态码速查

| 状态码 | 含义 | 常见场景 | 是否重试 |
|---:|---|---|---|
| `400` | 业务请求不正确 | 参数组合不允许 | 修改请求后重试 |
| `401` | 身份认证失败 | 密码错误、Token 无效或过期 | 重新登录 |
| `403` | 身份有效但权限不足 | 成员访问管理员接口 | 不应盲目重试 |
| `404` | 接口或资源不存在 | 地址错误、记录已删除 | 检查路径或编号 |
| `409` | 数据冲突 | 用户名重复、重复操作 | 修改冲突内容 |
| `422` | 参数格式错误 | 缺字段、类型或格式错误 | 修正字段后重试 |
| `429` | 请求过多 | 登录限流、外部平台限流 | 等待后重试 |
| `500` | 后端未知错误 | 未处理的代码异常 | 先查日志 |
| `502` | 外部平台失败 | AI 或音乐平台返回异常 | 根据错误码决定 |
| `503` | 依赖服务不可用 | PostgreSQL、Redis 不可用 | 服务恢复后重试 |
| `504` | 外部调用超时 | AI 或音乐平台超时 | 可以有限重试 |

## 5. 当前已实现错误码

| 错误码 | 状态码 | 含义 | 优先检查 |
|---|---:|---|---|
| `DATABASE_UNAVAILABLE` | `503` | PostgreSQL 无法正常执行查询 | Docker、PostgreSQL、`DATABASE_URL` |
| `VALIDATION_ERROR` | `422` | 请求字段缺失、类型错误或格式错误 | 请求 JSON、Schema、Swagger |
| `INTERNAL_SERVER_ERROR` | `500` | 后端发生未预料异常 | 根据 `request_id` 查堆栈 |
| `HTTP_404` | `404` | 接口或资源不存在 | URL、API 前缀、Router 挂载 |
| `HTTP_405` | `405` | 请求方法不允许 | GET、POST、PATCH、DELETE 是否正确 |
| `HTTP_401` | `401` | FastAPI 或依赖拒绝身份 | Token 和认证依赖 |
| `HTTP_403` | `403` | FastAPI 或依赖拒绝权限 | 用户角色和 Agent 权限 |

`HTTP_<状态码>` 是通用 HTTP 错误。业务模块优先使用含义更明确的错误码。

## 6. P2 已实现错误码

| 错误码 | 状态码 | 含义 |
|---|---:|---|
| `AUTH_INVALID_CREDENTIALS` | `401` | 账号或密码错误 |
| `AUTH_TOKEN_MISSING` | `401` | 未携带 Token |
| `AUTH_TOKEN_INVALID` | `401` | Token 签名或格式无效 |
| `AUTH_TOKEN_EXPIRED` | `401` | Token 已过期 |
| `AUTH_TOKEN_REVOKED` | `401` | 重置密码或停用账号后旧 Token 已失效 |
| `USER_INACTIVE` | `403` | 用户账号已停用 |
| `PERMISSION_DENIED` | `403` | 用户没有访问该功能的权限 |
| `USER_NOT_FOUND` | `404` | 用户不存在 |
| `USERNAME_ALREADY_EXISTS` | `409` | 登录账号已存在 |
| `AGENT_PERMISSION_DENIED` | `403` | 没有指定 Agent 使用权限 |
| `CANNOT_DISABLE_SELF` | `409` | 管理员试图停用自己的账号 |
| `PERMISSIONS_NOT_APPLICABLE` | `409` | 试图给超级管理员分配普通 Agent 权限 |

新增错误码后，必须同步更新本手册和自动化测试。

## 7. Agent 和外部平台错误码规则

后续模块按前缀分类：

```text
TASK_*         工作流任务
CRAWLER_*      榜单爬虫
ANALYSIS_*     分析Agent
LYRICS_*       作词Agent
MUSIC_*        音乐创作平台
EXTERNAL_*     其他第三方服务
CACHE_*        Redis
```

后台任务失败至少保存：

```text
task_id
error_code
error_message
failed_step
attempt_count
failed_at
```

外部服务错误还应记录平台名称、HTTP 状态码和平台请求编号，但不能记录 API Key。

## 9. 云服务器日志

生产环境建议设置：

```dotenv
ENVIRONMENT=production
LOG_LEVEL=INFO
LOG_FORMAT=json
LOG_TO_FILE=false
LOG_RETENTION_DAYS=30
```

此时 FastAPI 把 JSON 日志输出到标准输出，由 Docker 或云日志平台收集，不依赖 Windows 盘符。

查看后端最近日志：

```bash
docker compose logs --since 30m backend
```

按请求编号查找：

```bash
docker compose logs backend | grep "3f564950a27c45c39b00ec853c5979cd"
```

查看容器状态：

```bash
docker compose ps
```

查看服务器磁盘：

```bash
df -h
docker system df
```

生产 Compose 需要配置 Docker 日志大小和文件数量限制，避免磁盘被日志占满。

## 10. 配置项

| 环境变量 | 开发默认值 | 生产建议 | 说明 |
|---|---|---|---|
| `LOG_LEVEL` | `INFO` | `INFO` | 最低日志等级 |
| `LOG_FORMAT` | `text` | `json` | 日志格式 |
| `LOG_TO_FILE` | `true` | `false` | 是否同时写本地文件 |
| `LOG_DIR` | `backend/logs` | 按部署环境决定 | 文件日志目录 |
| `LOG_RETENTION_DAYS` | `30` | `30` | 文件日志保留天数 |

如果生产环境确实需要文件日志，应把 `LOG_DIR` 配置为容器挂载目录，不能依赖容器内部临时文件。

## 11. 常见问题排查

### 11.1 后端无法启动，没有 request_id

请求还没进入 FastAPI 时不会产生 `request_id`。优先查看启动终端最早出现的错误。

检查 Python 导入：

```powershell
D:\DevTools\Venvs\blue-music-backend\Scripts\python.exe -c "from app.main import app; print(app.title)"
```

检查环境变量、Python 依赖和端口占用。

### 11.2 DATABASE_UNAVAILABLE

检查容器：

```powershell
docker compose ps
```

检查 PostgreSQL：

```powershell
docker compose exec -T postgres pg_isready
```

预期输出包含：

```text
accepting connections
```

再检查 `.env` 中的 `DATABASE_URL` 是否指向正确主机和端口。

### 11.3 Redis 不可用

```powershell
docker compose exec -T redis redis-cli ping
```

预期输出：

```text
PONG
```

### 11.4 HTTP_404

按顺序检查：

```text
请求地址是否写错
是否包含 /api/v1 前缀
小路由是否定义
小路由是否挂到 api/v1/router.py
api_router 是否挂到 main.py
请求方法是否正确
```

### 11.5 VALIDATION_ERROR

查看 `detail.location`：

```text
body.username    JSON请求体中的username
query.page       URL查询参数page
path.user_id     URL路径中的user_id
```

再对照对应的 Pydantic Schema 检查字段名称、类型和是否必填。

### 11.6 INTERNAL_SERVER_ERROR

接口不会返回内部异常详情。必须复制 `request_id`，在日志中查找同一编号下面的 Python 堆栈。

从堆栈最底部开始看：

```text
异常类型
异常信息
最后一个属于本项目的文件
对应行号
```

## 12. 数据库错误后的 rollback

数据库操作抛出异常后，SQLAlchemy Session 可能进入失败状态。`get_db()` 会执行：

```text
数据库操作失败
-> rollback()
-> 继续抛出异常
-> 全局异常处理器记录日志
-> finally关闭Session
```

`rollback()` 只撤销当前未提交事务，不会删除已经正常提交的数据。

## 13. 安全规则

错误响应和普通日志禁止记录：

```text
明文密码
password_hash
Authorization请求头
Access Token和Refresh Token
Cookie
SECRET_KEY
第三方API Key
数据库连接密码
完整请求体
包含敏感值的SQL参数
```

SQLAlchemy 已启用 `hide_parameters=True`，数据库异常不会打印实际 SQL 参数值。

参数校验错误只返回位置、说明和错误类型，不回显用户原始输入。

## 16. 维护规则

每增加一种可预期错误，都要完成：

```text
1. 使用稳定、明确的错误码
2. 选择正确的HTTP状态码
3. 给用户安全、可理解的中文提示
4. 不把敏感信息放进detail
5. 增加正常路径和失败路径测试
6. 更新本手册的错误码表
```

不要为了方便直接把捕获到的异常文本返回给前端：

```python
# 禁止
return {"error": str(exc)}
```

未知异常统一返回 `INTERNAL_SERVER_ERROR`，真实原因只进入后端日志。
