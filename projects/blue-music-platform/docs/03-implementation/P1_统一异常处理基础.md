# P1 统一异常处理基础

本文记录 P1 阶段新增的统一异常处理基础。

## 1. 为什么需要统一异常处理

如果每个接口各自返回错误，前端就必须兼容多种格式：

```json
{"detail": "database connection is unavailable"}
```

```json
{"message": "not found"}
```

```json
{"error": "permission denied"}
```

统一异常处理的目标是让失败响应稳定、可预测。前端只需要读取同一套字段，后端也可以用错误码区分问题类型。

## 2. 当前错误响应格式

统一错误响应结构：

```json
{
  "error": {
    "code": "DATABASE_UNAVAILABLE",
    "message": "database connection is unavailable"
  }
}
```

字段含义：

- `code`：机器可读的错误码，方便前端和日志定位问题。
- `message`：用户或开发者可读的错误说明。
- `detail`：可选字段，用于放参数校验等结构化细节。

## 3. 涉及文件

```text
backend/app/schemas/errors.py
backend/app/core/exceptions.py
backend/app/core/exception_handlers.py
backend/app/main.py
backend/app/api/v1/routes/health.py
backend/tests/test_health.py
```

职责说明：

- `schemas/errors.py` 定义统一错误响应格式。
- `core/exceptions.py` 定义项目自己的 `AppException`。
- `core/exception_handlers.py` 定义 FastAPI 捕获异常后如何返回 JSON。
- `main.py` 注册全局异常处理器。
- `routes/health.py` 在数据库不可用时抛出 `AppException`。
- `tests/test_health.py` 验证数据库失败和 404 都返回统一错误格式。

## 4. 请求失败时的流程

数据库健康检查失败流程：

```text
GET /api/v1/health/database
-> routes/health.py
-> Depends(get_db)
-> services/health.py
-> is_database_available(db) 返回 False
-> routes/health.py 抛出 AppException
-> exception_handlers.py 捕获 AppException
-> 返回统一错误 JSON
```

404 路径不存在流程：

```text
GET /api/v1/missing
-> FastAPI 找不到路由
-> 抛出 HTTPException 404
-> exception_handlers.py 捕获 HTTPException
-> 返回统一错误 JSON
```

- 登录错误。
- 权限错误。
- Agent 任务错误。
- 外部 AI 或 Suno 错误。
- 复杂日志系统。
