# P6.5 线性自动工作流维护说明

## 1. 当前能力

用户可以在“自动流程”页面选择并保存以下步骤：

```text
榜单采集 -> 内容分析 -> 歌词创作
```

步骤可以减少，但必须保持数据依赖顺序。自动作词必须包含内容分析；音乐创作等待 P7 正式接口后再开放。

## 2. 数据传递

工作流执行器统一负责传参，Agent Service 之间不直接互相调用：

```text
CollectionTask.snapshot_id
-> 查询该快照前 30 首 RankingEntry.id
-> AnalysisCreateRequest.entry_ids

AnalysisReport.id
-> LyricsCreateRequest.analysis_report_id
-> 读取 direction_index 对应创作方向

LyricsVersion.id
-> P7 CreationBrief
-> MusicProvider 参数（待接入）
```

因此“步骤完成”必须同时得到可用的 `output_id`。上游失败或缺少产出时，流程立即失败并保留下游等待状态。

## 3. 数据表

```text
workflow_templates   保存名称、所选步骤和参数
workflow_runs       保存一次运行的配置快照和总状态
workflow_run_steps  保存每一步的 task_id、output_id、状态和错误
```

模板被修改后不会改变已经创建的运行；运行保存的是启动时配置快照。

## 4. 请求路线

```text
WorkflowsPage
-> frontend/src/api/workflows.ts
-> routes/workflows.py
-> services/workflows.py
-> rankings / analysis / lyrics Service
-> PostgreSQL
```

启动接口返回 HTTP 202，后台继续执行。前端检测到 `pending` 或 `running` 后每 2.5 秒刷新运行列表。

## 5. 运行约束

- 全系统同一时间只启动一条自动流程。
- 分析到作词默认冷却 15 秒，使用 `WORKFLOW_STEP_DELAY_SECONDS` 调整。
- 超过 `WORKFLOW_STALE_SECONDS` 的活动流程会被标记为 `WORKFLOW_TASK_INTERRUPTED`。
- 当前后台任务运行在 FastAPI 进程内；服务重启会中断任务。正式部署并发增加后再迁移到 Redis 队列 Worker。
- 手工运行分析或作词仍是同步请求，不属于自动流程后台调度。

## 6. 常用接口

```text
GET    /api/v1/workflows/templates
POST   /api/v1/workflows/templates
PUT    /api/v1/workflows/templates/{id}
DELETE /api/v1/workflows/templates/{id}
POST   /api/v1/workflows/templates/{id}/runs
GET    /api/v1/workflows/runs
DELETE /api/v1/workflows/runs
GET    /api/v1/workflows/runs/{id}
DELETE /api/v1/workflows/runs/{id}
```

运行历史支持单条和批量删除。删除 `workflow_runs` 时会级联删除对应的 `workflow_run_steps`，但不会删除流程模板，也不会删除各步骤已经生成的榜单、分析或歌词产出。`pending` 和 `running` 状态不可删除；批量请求中存在无效编号或活动流程时整批拒绝。

## 7. 测试重点

`backend/tests/test_workflows.py` 覆盖：

- 采集快照真实传给分析任务。
- 分析报告真实传给作词任务。
- 三步运行均保存任务编号和产出编号。
- 上游缺数据时停在失败步骤。
- 作词不能脱离分析单独加入自动流程。
- 运行历史批量删除会去重编号，并保留流程模板和步骤产出。
- 排队或运行中的流程不能删除。

前端 `frontend/src/lib/workflows.test.ts` 覆盖步骤依赖联动。
