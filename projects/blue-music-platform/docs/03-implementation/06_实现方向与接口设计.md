# 蓝乐 AI 音乐创作平台 — 实现方向与接口设计分析

> 本文档对 SRS 中每个模块的实现方向、技术难点、API 接口设计进行详细分析。
> 最后更新：2026-05-27

## 一、各模块实现方向分析

### 模块1：用户管理（FR-0101 ~ FR-0103）

**实现方向：**
- 后端：FastAPI + SQLAlchemy + bcrypt 密码哈希
- 前端：React 表单组件（Ant Design Form）
- JWT 生成：python-jose 库，access_token 24h，refresh_token 7天

**技术难点：**
- Token 刷新机制：前端拦截 401 → 自动用 refresh_token 换新 access_token
- 密码强度校验：正则表达式（至少8位，含字母数字）

**数据流：**
```
注册：前端表单 → POST /api/auth/register → bcrypt哈希 → 存入users表 → 返回user_id
登录：前端表单 → POST /api/auth/login → 验证密码 → 生成JWT → 返回token
```

---

### 模块2：权限管理 RBAC（FR-0201 ~ FR-0203）

**实现方向：**
- 数据库表：roles、permissions、role_permissions、user_roles
- 中间件：FastAPI Depends() 注入权限校验
- 前端：菜单根据权限动态渲染

**技术难点：**
- 权限粒度设计：接口级权限（如 `crawler:run`），不是页面级
- 权限缓存：用户登录后权限存入 Redis，避免每次请求查数据库

**权限列表设计：**
```
user:list      — 查看用户列表
user:create    — 创建用户
user:edit      — 编辑用户
user:delete    — 删除用户
role:create    — 创建角色
role:edit      — 编辑角色
agent:create   — 创建Agent
agent:edit     — 编辑Agent配置
crawler:run    — 执行爬虫
analyze:run    — 执行分析
create:run     — 执行创作
backup:manage  — 管理备份
backup:restore — 执行回档
```

**数据流：**
```
管理员创建角色 → 勾选权限 → 存入role_permissions表
管理员分配角色给用户 → 存入user_roles表
用户登录 → 加载权限列表 → 存入Redis → 前端根据权限显示菜单
每次API请求 → 中间件校验权限 → 无权限返回403
```

### 模块3：Agent 热插拔框架（FR-0301 ~ FR-0303）★核心

**实现方向：**
- 每个 Agent 是一个 Python 类，继承 BaseAgent
- Agent 配置存储在数据库 agents 表
- 运行时从数据库加载配置，动态实例化

**BaseAgent 设计：**
```python
class BaseAgent:
    def __init__(self, config: dict):
        self.agent_type = config['agent_type']
        self.model_name = config['model_name']
        self.api_key = decrypt(config['api_key_encrypted'])
        self.temperature = config['temperature']
        self.status = config['status']

    def run(self, prompt: str) -> dict:
        raise NotImplementedError
```

**子类实现：**
```python
class CrawlerAgent(BaseAgent):
    def run(self, prompt=None):
        # 爬虫逻辑
        pass

class AnalyzerAgent(BaseAgent):
    def run(self, prompt):
        # 调用LLM分析
        pass

class CreatorAgent(BaseAgent):
    def run(self, prompt):
        # 调用Suno API
        pass
```

**热插拔机制：**
- 启动时：从数据库加载所有 enabled 的 Agent → 实例化 → 存入内存字典
- 配置变更：更新数据库 → 发送信号 → Agent 管理器重新加载该 Agent
- 模型切换：修改 model_name → 下次调用时使用新模型的 API 端点

**技术难点：**
- API Key 加密存储：AES-256，密钥从环境变量读取
- 多模型适配：OpenAI / Claude / 文心 各有不同的 API 格式，需要适配层
- 运行时重载：不能影响正在执行的任务

### 模块4：爬虫 Agent（FR-0401 ~ FR-0403）

**实现方向：**
- 技术选型：Playwright（比 Selenium 更轻量，支持无头模式）
- 定时任务：APScheduler（Python 定时任务库）
- 数据存储：PostgreSQL，去重逻辑在插入时处理

**爬取流程：**
```
1. Playwright 启动无头浏览器
2. 打开 QQ音乐热歌榜页面
3. 等待页面加载，滚动到底部加载全部500条
4. 解析HTML，提取：歌曲名、歌手、专辑、播放量、排名
5. 数据清洗（去除空格、特殊字符）
6. 批量插入数据库（ON CONFLICT DO NOTHING 去重）
7. 记录爬取日志
```

**反反爬策略：**
- 随机 User-Agent（从预置列表随机选）
- 请求间隔 1-3 秒随机
- 代理 IP 池（预留接口，可后续接入付费代理）

**技术难点：**
- QQ音乐页面可能是动态渲染的（JS加载），必须用 Playwright 而非 requests
- 榜单页面结构可能变化，需要定期维护选择器
- 代理 IP 的质量和稳定性

### 模块5：分析 Agent（FR-0501 ~ FR-0503）

**实现方向：**
- 调用 OpenAI GPT-4o API（Function Calling 模式）
- 结构化输出：强制返回 JSON 格式
- 批量处理：异步队列（Celery 或 FastAPI BackgroundTasks）

**Prompt 设计：**
```
你是一个音乐风格分析专家。请分析以下歌曲：

歌曲：《{song_name}》
歌手：{artist}
专辑：{album}

请以JSON格式返回分析结果：
{
  "style_tags": ["流行", "摇滚", ...],  // 曲风标签
  "emotion": "欢快",                    // 情绪
  "bpm_range": "80-120",               // BPM范围
  "language": "中文",                   // 语种
  "keywords": ["爱情", "青春", ...]     // 关键词
}
```

**技术难点：**
- LLM 返回格式不稳定：需要重试机制 + JSON 解析容错
- 批量分析成本控制：500首歌 × 每首约500 token = 25万 token，约 $0.5-1
- 分析结果缓存：同一首歌不重复分析

### 模块6：创作 Agent（FR-0601 ~ FR-0603）

**实现方向：**
- 对接 Suno API（HTTP POST）
- 自动生成 Prompt：根据分析结果拼接风格标签
- 异步任务：创作耗时较长（30秒-2分钟），用后台任务处理

**Prompt 生成逻辑：**
```python
def generate_prompt(analysis_result):
    style = "、".join(analysis_result['style_tags'])
    emotion = analysis_result['emotion']
    bpm = analysis_result['bpm_range']
    keywords = "、".join(analysis_result['keywords'])
    
    return f"一首{emotion}的{style}歌曲，BPM约{bpm}，主题关于{keywords}"
```

**Suno API 调用（示例）：**
```python
response = httpx.post(
    "https://api.suno.ai/v1/generate",
    headers={"Authorization": f"Bearer {api_key}"},
    json={
        "prompt": prompt,
        "duration": 30,  # 30秒片段
        "style": style_tags[0]
    }
)
```

**技术难点：**
- Suno API 可能不稳定，需要重试和超时处理
- 音频文件存储：本地 / OSS / 直接引用 Suno CDN
- 批量创作并发控制：避免同时发起太多请求被限流

### 模块8：系统备份与回档（FR-0801 ~ FR-0804）

**实现方向：**
- 备份：PostgreSQL pg_dump 命令
- 定时：APScheduler 每日凌晨执行
- 加密：备份文件用 AES-256 加密后存储
- 回档：pg_restore 命令恢复

**备份流程：**
```
1. APScheduler 触发定时任务
2. 执行 pg_dump -Fc（自定义格式，支持选择性恢复）
3. 压缩备份文件
4. AES-256 加密
5. 存储到指定位置（本地/OSS）
6. 记录备份日志
7. 清理7天前的旧备份
```

**回档流程：**
```
1. 管理员选择备份点
2. 系统预览影响范围（哪些表、多少条记录）
3. 管理员输入密码二次确认
4. 解密备份文件
5. 执行 pg_restore
6. 验证数据一致性
7. 重启服务
8. 生成恢复报告
```

**技术难点：**
- 回档是高风险操作，必须做好安全防护
- 大数据库备份耗时，需要异步执行
- 回档后服务重启的平滑处理

## 三、统一响应格式

所有接口统一使用以下格式：

```json
{
  "code": 0,        // 0=成功，其他=错误码（见SRS附录B）
  "message": "xxx", // 提示信息
  "data": {}        // 业务数据
}
```

## 四、错误处理

| 场景 | HTTP状态码 | 错误码 | 处理方式 |
|------|-----------|--------|----------|
| 参数校验失败 | 400 | 1001 | 返回具体字段错误 |
| Token过期 | 401 | 1002 | 前端自动刷新Token |
| 权限不足 | 403 | 1003 | 提示无权限 |
| 资源不存在 | 404 | 1004 | 返回资源不存在 |
| 业务逻辑错误 | 400 | 2001 | 返回具体业务错误 |
| 第三方API失败 | 502 | 3001 | 重试3次后报错 |
| 服务器内部错误 | 500 | 5000 | 记录日志，返回通用错误 |
