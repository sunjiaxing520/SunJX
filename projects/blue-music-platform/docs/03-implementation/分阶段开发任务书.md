# 蓝乐 AI 音乐创作平台分阶段开发任务书

版本：v1.7
制定日期：2026-07-15
依据：需求规格说明书 v3.5、项目决策记录

## 1. 项目目标

首期交付一个清楚、可操作、可演示、可验收的 AI 音乐创作工作台，并支持既有 Agent 的线性自动编排。

完整业务闭环：

```text
登录
-> 选择或启动已保存的线性工作流
-> 酷狗榜单采集
-> 榜单趋势分析
-> AI 作词
-> 官方音乐生成 API
-> Dashboard 和结果展示
```

首期采用“一次点击后，按数据依赖自动执行多个独立 Agent”的模式。

首期不包含：

- 拖拽式任意工作链、条件分支、并行节点和循环。
- 总调度 AI 自动理解目标和拆分任务。
- 多 Agent 自动协作。
- 复杂长期记忆系统。
- 多音乐平台数据源。
- 自研音乐生成模型。

## 2. 执行原则

1. 每个阶段必须完成代码、测试、文档和讲解后才能验收。
2. 上一阶段未通过验收，不开始依赖它的下一阶段。
3. 每个功能先跑通最小闭环，再增加细节。
4. 外部服务必须通过适配层接入，业务代码不能直接绑定单一供应商。
5. 密钥和密码不写死在代码中；启动后需热切换的供应商密钥使用后端加密存储，环境变量保留为初始化和故障后备。
6. 每次开发都要说明请求从哪里进入、经过哪些文件、数据如何保存、结果如何返回。

## 11. P7：音乐创作 Agent

对应 SRS：FR-006。

### 开发任务

- 设计创作任务和创作结果数据模型。
- 收集标题、创作类型、歌词、风格、情绪、节奏、人声和补充要求。
- 建立统一 `MusicProvider` 接口和一个正式供应商适配器；供应商支持时暴露可选的续写能力。
- 实现任务提交、状态查询、失败处理和结果回收。
- 保存外部任务编号、结果链接和错误信息。
- 将音乐生成、查询和续写调用写入统一 API 用量账本，保存真实计费单位和供应商请求编号。
- 统一结果字段至少包含音频地址、下载地址、格式、时长和供应商结果编号。
- 实现创作任务列表和详情页。
- 实现独立试听区，按生成完成时间罗列所有可播放结果，点击某一首时使用该结果的音频地址播放。
- 首段初稿和续写结果都作为真实生成结果进入试听区；播放不向音乐供应商提交新任务。
- 对具备续写能力的结果提供“继续续写”操作，由用户试听满意后主动创建独立续写任务。
- 续写任务保存来源结果、供应商歌曲编号、续写位置和新增歌词，不覆盖原始结果。
- 每个生成音频支持独立播放和独立下载。
- 下载时优先调用浏览器保存文件选择器，让用户选择目标位置；不支持时退回浏览器默认下载。
- 为自动化测试提供明确的模拟适配器，不伪装为真实生成。

### 学习目标

- 理解业务任务和第三方任务编号的区别。
- 理解轮询、回调、超时和状态机。
- 理解适配层如何降低第三方平台变化的影响。
- 理解为什么不能在前端暴露第三方 API Key。

### 交付物

- 创作任务和结果模型，包括续写任务与来源结果的关联。
- `MusicProvider` 接口、正式供应商适配器和模拟适配器。
- 创建任务、查询状态和保存结果的接口与页面。
- 外部服务异常处理和测试。

### 完成标准

- 用户可以从空白表单或已保存歌词创建创作任务。
- 任务状态至少支持排队中、执行中、完成和失败。
- 完成任务能够保存并展示结果链接。
- 同一任务返回的每个音频都会进入试听区，能单独点击播放，并能单独选择保存位置下载。
- 支持续写的供应商能够从指定初稿创建续写任务；初稿和续写结果可分别播放、下载和追踪状态。
- 失败任务能够展示可理解的失败原因。
- API Key 只存在于后端环境配置或数据库加密字段中，永不返回前端明文。
- 切换模拟适配器和真实适配器不需要改动核心业务流程。

## 13. 项目总体验收标准

项目达到首期交付条件时，必须同时满足：

### 功能

- FR-001 至 FR-009 的首期验收项全部通过。
- 完整业务闭环可以现场演示。
- 外部服务不可用时，系统能说明原因且不丢失已有数据。

### 安全

- 密码加密存储。
- 后台接口要求有效身份。
- 超级管理员接口有后端权限校验。
- 密钥不进入 Git 仓库，也不暴露给前端。

### 质量

- 核心流程有自动化测试。
- 数据库迁移可以在新数据库执行。
- 任务失败有状态和错误记录。
- 代码目录职责清楚，没有把全部逻辑堆在路由文件中。

### 运维

- Docker Compose 可以启动必要服务。
- 有环境变量说明、部署步骤和备份建议。
- 有日志查看和常见故障排查说明。

### 学习和维护

- 每个阶段都有对应代码流程说明。
- 开发者能独立启动项目并使用 Swagger 测试接口。
- 开发者能说明前端请求如何经过路由、Service、数据库或外部服务再返回。
- 开发者能根据日志定位常见的参数、权限、数据库和外部 API 问题。
