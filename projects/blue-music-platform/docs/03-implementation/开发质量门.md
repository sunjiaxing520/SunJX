# 蓝乐项目开发质量门

本文档定义每个开发增量合并和进入下一阶段前必须满足的条件。

## 1. 开发顺序

每个功能统一按以下顺序推进：

```text
确认需求和验收标准
-> 画出请求与数据流程
-> 先写失败测试或测试场景
-> 编写最小实现
-> 运行自动化测试
-> 人工验证关键流程
-> 讲解代码和记录决策
-> 检查差异
-> 提交并推送
```

## 2. 开始开发前

- 已执行 `git status` 和 `git pull`。
- 工作区没有来源不明且会影响本任务的修改。
- 当前任务能对应到 SRS、决策记录或明确的技术底座任务。
- 已写明本次不做什么，防止范围扩大。
- 已确定可客观判断的完成标准。

## 3. 代码完成标准

- 文件职责清楚，路由文件不堆积业务和数据库逻辑。
- 外部配置来自环境变量，密钥不进入仓库。
- 输入和输出使用明确的 Schema 校验。
- 错误不会被静默吞掉，用户能得到可理解的信息。
- 数据库修改通过迁移完成，不手工修改生产表结构。
- 第三方服务通过适配层接入。
- 只修改当前任务需要的文件。

## 4. 测试完成标准

- 新增行为有自动化测试。
- 正常路径、关键失败路径和权限路径均有验证。
- 测试不依赖真实付费 API；第三方服务使用模拟适配器。
- 所有已有测试通过。
- 必要时完成浏览器人工验证和接口人工验证。

后端基础测试命令：

```powershell
cd projects\blue-music-platform\backend
..\.venv\Scripts\python -m pytest
```

## 5. 学习完成标准

每个增量完成后，开发者需要能回答：

1. 用户或前端发出了什么请求？
2. 请求首先进入哪个文件？
3. Schema、Service、Model 分别做什么？
4. 数据保存在哪里？
5. 成功和失败分别返回什么？
6. 哪些测试证明代码符合要求？
7. 将来修改这个功能，最可能改哪些文件？

如果这些问题无法回答，本阶段只能算代码完成，不能算学习完成。

## 6. 提交完成标准

- 已执行 `git diff` 检查实际改动。
- 未提交 `.env`、虚拟环境、缓存、密钥或生成垃圾文件。
- 提交信息准确描述单一改动。
- `git push` 成功。
- 推送后 `git status` 显示与远端同步且工作区干净。

## 7. 阶段门

阶段验收结论只有三种：

- 通过：全部完成标准满足，可以进入下一阶段。
- 有条件通过：不影响主链路的问题已记录，并明确补齐时间。
- 不通过：关键功能、测试、安全或学习标准未满足，不进入下一阶段。
