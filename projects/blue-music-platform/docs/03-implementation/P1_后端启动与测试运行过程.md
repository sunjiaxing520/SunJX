# P1 后端启动与测试运行过程

最后更新：2026-07-05

本篇只解决一个问题：代码你已经看懂了，但它到底是怎么“跑起来”的。

这里要分清两件事：

```text
真实启动后端：给浏览器、前端、外部请求访问。
运行测试：在 Python 进程里模拟请求，不需要真的打开端口。
```

## 1. 真实后端是怎么启动的

启动命令：

```powershell
cd D:\SunJX\projects\blue-music-platform\backend
D:\DevTools\Venvs\blue-music-backend\Scripts\python.exe -m uvicorn app.main:app --host 127.0.0.1 --port 8000
```

这条命令可以拆开理解。

```text
D:\DevTools\Venvs\blue-music-backend\Scripts\python.exe
```

意思是：使用项目自己的 Python 环境。

这个环境里安装了 FastAPI、Uvicorn、SQLAlchemy、pytest 等依赖，不用系统 Python 随便找。

```text
-m uvicorn
```

意思是：用 Python 启动 `uvicorn` 这个模块。

Uvicorn 是后端服务器。它负责监听端口，接收浏览器或前端发来的 HTTP 请求。

```text
app.main:app
```

这是最关键的一段。

它的意思是：

```text
导入 app/main.py 文件
从这个文件里拿到名字叫 app 的对象
```

对应文件：

```text
D:\SunJX\projects\blue-music-platform\backend\app\main.py
```

对应代码：

```python
app = create_app()
```

也就是说，Uvicorn 真正启动的不是整个目录，而是 `app/main.py` 里的这个 `app` 对象。

```text
--host 127.0.0.1
```

意思是：只允许本机访问。

```text
--port 8000
```

意思是：后端监听 8000 端口。

所以浏览器访问：

```text
http://127.0.0.1:8000/api/v1/health
```

请求才会进入这个后端。

## 2. 真实请求的完整运行过程

以这个地址为例：

```text
GET http://127.0.0.1:8000/api/v1/health
```

运行链路是：

```text
浏览器
-> 127.0.0.1:8000
-> Uvicorn
-> app.main:app
-> create_app()
-> app.include_router(api_router, prefix="/api/v1")
-> api/v1/router.py
-> routes/health.py
-> health_check()
-> HealthResponse
-> JSON
-> 浏览器
```

重点是：

```text
Uvicorn 负责“接电话”。
FastAPI app 负责“找接口”。
routes/health.py 里的函数负责“处理这次请求”。
schemas/health.py 负责“规定返回格式”。
```

## 3. main.py 什么时候执行

很多初学者会误以为 `main.py` 只有浏览器访问时才执行。

实际不是。

当 Uvicorn 看到：

```text
app.main:app
```

它会先导入：

```python
from app.main import app
```

导入这个文件时，Python 会执行文件最外层的代码。

所以这句会执行：

```python
app = create_app()
```

这时 FastAPI 应用已经创建好了，路由也已经挂载好了。

浏览器后续访问时，不是重新创建 app，而是使用这个已经创建好的 app 来处理请求。

## 4. pytest 为什么可以直接跑

测试命令：

```powershell
cd D:\SunJX\projects\blue-music-platform\backend
D:\DevTools\Venvs\blue-music-backend\Scripts\python.exe -m pytest -vv
```

这条命令没有写测试文件名，但它仍然知道测哪里，原因在：

```text
D:\SunJX\projects\blue-music-platform\backend\pytest.ini
```

内容是：

```ini
[pytest]
testpaths = tests
python_files = test_*.py
python_functions = test_*
addopts = -q
```

这几行的意思：

```text
testpaths = tests
只去 tests 目录找测试。

python_files = test_*.py
只把 test_ 开头的 Python 文件当测试文件。

python_functions = test_*
只把 test_ 开头的函数当测试函数。

addopts = -q
默认使用简洁输出。
```

所以你运行：

```powershell
python -m pytest
```

pytest 会自动找到：

```text
tests/test_health.py
```

然后自动找到里面这些函数：

```text
test_root_returns_running_message
test_health_returns_service_metadata
test_database_health_returns_healthy_when_query_succeeds
test_database_health_returns_503_when_query_fails
test_not_found_returns_unified_error_response
```

这就是“为什么直接就能跑”。

不是它猜到了，而是 `pytest.ini` 已经告诉它规则了。

## 5. 测试时有没有启动 Uvicorn

没有。

测试不是这样跑的：

```text
pytest -> 启动 uvicorn -> 打开 8000 端口 -> 发真实 HTTP 请求
```

测试实际是这样跑的：

```text
pytest
-> 导入 tests/test_health.py
-> 导入 app.main 里的 app
-> conftest.py 创建 TestClient(app)
-> client.get("/api/v1/health")
-> 在内存里模拟 HTTP 请求
-> FastAPI 正常走路由匹配
-> 返回响应对象
-> assert 检查结果
```

对应文件：

```text
D:\SunJX\projects\blue-music-platform\backend\tests\conftest.py
```

关键代码：

```python
@pytest.fixture
def client() -> TestClient:
    return TestClient(app)
```

这里的 `TestClient(app)` 就是测试的关键。

它把 FastAPI 的 `app` 包起来，让测试代码可以写：

```python
response = client.get("/api/v1/health")
```

这行看起来像浏览器请求，但它不经过网卡、不占用端口、不需要 Uvicorn。

它是在 Python 进程内部直接调用 FastAPI 应用。

## 6. 测试和真实启动的区别

| 项目 | 真实启动 | 运行测试 |
|---|---|---|
| 启动命令 | `python -m uvicorn app.main:app` | `python -m pytest` |
| 是否需要端口 | 需要，当前是 8000 | 不需要 |
| 是否需要浏览器 | 可以用浏览器访问 | 不需要 |
| 请求从哪里来 | 浏览器、前端、外部工具 | `TestClient` |
| 是否经过 Uvicorn | 是 | 否 |
| 是否使用同一个 FastAPI app | 是 | 是 |
| 是否走同一套路由代码 | 是 | 是 |

所以测试虽然没有启动服务器，但它测的仍然是同一个 `app`、同一批路由、同一批接口函数。

## 7. 数据库测试为什么不用真数据库

数据库健康检查接口里有：

```python
def database_health_check(db: Session = Depends(get_db)) -> DatabaseHealthResponse:
```

正常运行时，`Depends(get_db)` 会提供真实数据库连接。

但测试里用了：

```python
app.dependency_overrides[get_db] = override_get_db
```

意思是：

```text
这次测试里，如果代码需要 get_db，不要调用真的 get_db。
改用 override_get_db。
```

成功测试里，替换成：

```python
class WorkingSession:
    def execute(self, statement: object) -> None:
        return None
```

失败测试里，替换成：

```python
class FailingSession:
    def execute(self, statement: object) -> None:
        raise SQLAlchemyError("database unavailable")
```

所以测试可以分别模拟：

```text
数据库正常
数据库失败
```

这比真的连接数据库更快、更稳定，也更容易验证错误逻辑。

## 8. 这次测试输出怎么看

实际运行：

```powershell
D:\DevTools\Venvs\blue-music-backend\Scripts\python.exe -m pytest -vv
```

关键输出：

```text
rootdir: D:\SunJX\projects\blue-music-platform\backend
configfile: pytest.ini
testpaths: tests
collected 5 items
5 passed
```

逐句解释：

```text
rootdir
pytest 认为当前后端项目根目录是哪里。

configfile
pytest 读取了哪个配置文件。

testpaths
pytest 根据配置去哪个目录找测试。

collected 5 items
pytest 找到了 5 个测试函数。

5 passed
5 个测试全部通过。
```

## 9. 你需要能复述的版本

最短版本：

```text
真实运行后端时，Uvicorn 根据 app.main:app 导入 app/main.py 里的 app 对象，
然后监听 127.0.0.1:8000，浏览器请求进来后交给 FastAPI 匹配路由。

运行测试时，pytest 根据 pytest.ini 自动找到 tests/test_health.py，
测试文件导入同一个 app，然后 conftest.py 用 TestClient(app) 在内存里模拟请求。

所以测试不用启动 Uvicorn，也不用打开浏览器，但仍然测的是同一套接口代码。
```

如果你能讲出这段，第一部分的“运行过程”就算真正理解了。
