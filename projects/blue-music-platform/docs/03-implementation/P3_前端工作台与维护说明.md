## 3. 关键运行路线

```text
浏览器打开前端
-> AuthProvider 检查本地 Token
-> GET /api/v1/auth/me 恢复当前用户
-> ProtectedRoute 决定能否进入工作台
-> AppShell 根据角色和 Agent 权限生成导航
-> DashboardPage 请求 GET /api/v1/dashboard
-> 后端只返回当前用户有权看到的 Agent
```

登录路线：

```text
LoginPage
-> POST /api/v1/auth/login
-> 保存 Access Token
-> GET /api/v1/auth/me
-> 进入 Dashboard
```

密码重置、账号停用、Token 过期或 Token 被撤销后，后端返回对应认证错误码，前端会自动清理登录状态并回到登录页。

## 4. Dashboard 数据口径

接口：`GET /api/v1/dashboard`。

当前 P3 尚无业务任务表，因此四项指标返回 `0`。Agent 执行器尚未在 P4 至 P7 接入，所以状态返回 `not_configured`。这是当前系统真实状态，不是模拟失败。

超级管理员可看到全部四类 Agent；成员只会看到管理员已分配的 Agent。

## 5. 错误定位

前端 API 请求统一经过 `frontend/src/api/client.ts`。错误对象包含：

```text
HTTP status
error code
中文 message
request_id
安全 detail
```

最近 20 条 API 或页面渲染错误保存在浏览器本地。右上角账号菜单选择“复制诊断信息”，即可生成维护报告。

报告不会包含：

- Access Token。
- 密码。
- 登录或其他请求正文。
- API Key。

快速判断：

| 现象 | 优先检查 |
|---|---|
| `NETWORK_ERROR` 且无 `request_id` | 后端是否启动、8000 端口、`VITE_API_BASE_URL`、CORS |
| 有 `request_id` | 在 `backend/logs/app.log*` 搜索该编号 |
| 自动返回登录页 | Token 过期、撤销、账号停用或认证失败 |
| 成员缺少菜单 | 检查超级管理员分配的 Agent 权限 |
| 前端能打开但无法登录 | 后端 `CORS_ORIGINS` 是否包含当前前端域名和端口 |

## 6. 配置与云部署边界

开发默认值：

```dotenv
VITE_API_BASE_URL=http://127.0.0.1:8000/api/v1
CORS_ORIGINS=http://localhost:5173,http://127.0.0.1:5173
```

云端不能继续使用 `127.0.0.1` 作为浏览器访问的 API 地址。生产构建时需要把 `VITE_API_BASE_URL` 设置为正式后端 HTTPS 地址，并把前端正式域名加入后端 `CORS_ORIGINS`。正式反向代理、HTTPS 和前端容器在 P8 部署阶段完成。
