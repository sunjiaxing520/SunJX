# 蓝乐 AI 音乐创作平台决策记录

## 2026-06-02：首期爬虫数据源

结论：基础版首期先爬 **酷狗热歌榜**。

原因：

- 报价方案中基础版已明确提到酷狗热歌榜。
- SRS 中曾出现 QQ 音乐描述，后续以本决策记录和报价方案为准。
- 后续如进入完整版，再扩展 QQ 音乐、网易云音乐等多源爬取。

开发影响：

- P1/P2 之后的爬虫 Agent 优先实现酷狗热歌榜数据采集。
- 数据模型仍保留 `platform` 字段，为后续多源扩展预留。
- 文档、任务说明和 Cursor 提示词中统一称为“酷狗热歌榜”。

## 2026-06-02：首期创作工具

状态：已被 2026-07-11“官方音乐生成 API”决策取代。

结论：初步暂定使用 **Suno** 作为首期音乐创作工具。

说明：

- 平台不直接承诺“完全原创替代人工作曲”，而是把用户输入和分析结果整理成更适合 Suno 使用的创作素材。
- 对客户讲解时尽量使用中文字段，不直接讲英文接口参数。
- 后续如果 Suno 接口或平台规则变化，应保留创作工具适配层，避免业务逻辑和单一平台强绑定。

首期创作需要准备的中文素材：

```text
歌曲标题
创作类型：带人声 / 纯音乐
歌词：用户填写或 AI 辅助生成
音乐风格：流行、摇滚、电子、民谣、R&B 等
情绪氛围：欢快、悲伤、舒缓、激昂等
节奏速度：慢速 / 中速 / 快速，或 BPM 范围
人声要求：男声 / 女声 / 不指定
补充要求：用户自由描述
```

完整版可扩展：

```text
歌曲结构：主歌 / 副歌 / 过渡段
不想要的风格
多个创作版本
在线试听
版本回退
```

## 2026-06-02：Agent 操作模式

状态：已被 2026-07-11“首期自动工作流和 Agent 分工”决策取代。

结论：首期先做 **人操作多个 Agent**，后续再升级为 **总调度 Agent 调用多个专业 Agent**。

首期模式：

```text
用户分别操作爬虫 Agent、分析 Agent、创作 Agent。
```

后续模式：

```text
用户只提出目标，总调度 Agent 自动拆任务并调用多个专业 Agent。
```

原因：

- 首期先保证客户能看懂、能操作、能验收。
- 人工操作模式开发范围更清晰，风险更低。
- 总调度 Agent 可以作为后续增强能力，不需要推翻现有多个专业 Agent 的设计。

边界说明：

- 首期 Agent 更偏“自动化任务机器人”，例如采集榜单、提交表单、跟踪任务、保存结果。
- 后续增强 Agent 才考虑引入更强的 AI 推理、长期记忆和自动调度能力。
- 自定义工作链、Agent 记忆系统、总调度 Agent 均放入初步设想，不纳入当前首期交付范围。

## 2026-06-02：新增作词 Agent

状态：保留作词 Agent，但基础记忆和多个作词 Agent 调整为后续增强范围。

结论：首期增加 **作词 Agent**。

原因：

- 客户希望用 AI 写歌词。
- Suno 创作带人声歌曲时，歌词是核心素材。
- 作词质量和客户偏好强相关，所以作词 Agent 需要基础记忆。

首期作词 Agent 的能力：

```text
根据主题生成歌词
根据风格生成歌词
根据情绪生成歌词
支持重新生成
支持局部修改副歌/主歌
支持把歌词送入 Suno 创作 Agent
```

首期基础记忆：

```text
用户常用主题
用户偏好的歌词风格
常用表达方式
不想要的词或风格
历史歌词记录
```

边界：

- 首期只做基础偏好记忆，不做复杂长期记忆系统。
- 多个作词 Agent、不同作词 Agent 拥有不同记忆，放入后续增强项目。
- 总调度 Agent 自动选择作词 Agent，也放入后续增强项目。

## 2026-07-11：首期自动工作流和 Agent 分工

结论：首期采用 **一条固定自动工作流**，用户点击一次后，系统按固定顺序自动执行各个 Agent。

```text
酷狗榜单采集
-> 榜单趋势分析
-> AI 作词
-> 官方音乐生成 API
-> 保存结果并展示
```

实现边界：

- 工作流调度器统一控制执行顺序，Agent 之间不直接互相调用。
- 爬虫、分析、作词、音乐生成仍保持独立模块，分别定义清楚输入和输出。
- 首期每种 Agent 只有一个默认实现，不做用户选择多个 Agent。
- 首期不做总调度 AI、自定义工作链、Agent 热插拔和复杂长期记忆。
- 每一步保存状态和结果，失败时记录原因，并允许重新执行。
- 前端使用轮询查看任务状态，首期不增加 WebSocket 和复杂实时系统。

后续扩展时，可以在同一工作流中为不同步骤选择不同的 Agent 实例，但不影响首期固定流程。

## 2026-07-11：内部团队、账号和数据范围

结论：首期是 **单客户内部团队工具**，不是面向公众开放注册的 SaaS。

- 系统不开放用户自助注册。
- 初始化一个超级管理员，由超级管理员创建、停用账号并分配 Agent 使用权限。
- 榜单、分析、歌词、任务和创作结果默认在团队内共享。
- 每条任务和结果仍记录创建人，便于追踪来源。
- 榜单按自然日保存快照，默认保留最近 30 天，页面最多展示最近 15 次运行记录。
- 同一天重复采集时更新当天快照，不伪造额外历史天数。
- 榜单分析默认使用连续 7 天快照；实际不足 7 天时按现有天数分析并降低证据可信度。
- 普通任务产物默认保留 30 天，后续实现时再确定主动保存作品是否长期保留。

## 2026-07-11：官方音乐生成 API

结论：首期不使用非官方 Suno 套壳接口或网页自动化，统一通过官方音乐生成 API 适配层接入。

当前候选顺序：

1. Mureka 官方 API：首选候选，支持中文歌词、完整歌曲、异步任务状态和商业授权。
2. MiniMax Music 官方 API：备选候选，支持中文歌词和完整歌曲生成。

开发要求：

- 在客户提供正式账号前，先用开发测试账号完成同一批中文歌曲对比测试。
- 根据中文咬字、音乐质量、生成时间、水印、价格和稳定性确定最终首期供应商。
- 业务模块只依赖统一的 `MusicProvider` 接口，不在核心流程中写死供应商名称。
- 首期只真正接入一个供应商，不做多供应商自动切换。
- API Key 只保存在后端环境变量中，不进入前端和 Git 仓库。

## 2026-07-13：分析和作词按音乐平台输入反向设计

结论：P5 和 P6 不只输出说明文章，而是共同生成可供正式音乐生成 API 使用的结构化创作方案。

- P5 输出曲风、情绪、主题、场景、速度档位、人声方向、乐器方向、结构和 Hook 建议。
- P6 输出标题、标准段落标签歌词和风格提示词。
- 不从榜单元数据假装推断准确 BPM、调性、和弦或真实编曲。
- P7 只负责把统一创作方案映射成最终供应商参数，避免业务流程绑定单一平台。
- 仓库默认使用明确标记的本地规则基线；正式文本模型通过 OpenAI-compatible 适配器和后端环境变量接入。

## 2026-07-15：开发阶段文本模型

结论：本机开发联调使用智谱免费模型 `glm-4.7-flash`，通过现有 OpenAI-compatible 适配器接入。API Key 只保存在被 Git 忽略的本机 `.env`，自动化测试和未配置环境继续使用本地规则基线。正式部署前仍需确认生产账号、额度、稳定性和商用条款。

## 2026-07-15：榜单结果详情与音乐文件操作

结论：榜单采集主页面不再常驻展示整张排行榜，改为“折叠的采集结果块 -> 点击结果 -> 在块内展开快照详情”，且一次只展开一个快照。运行记录继续独立保留，用于查看等待、运行、完成、失败和错误原因。

P7 音乐结果必须满足：

- 同一任务返回多个音频时逐首展示，不把多个结果合并成一个下载项。
- 系统建立独立试听区，统一罗列已完成的音乐结果；点击某一首后播放该结果的完整音频。
- 首段初稿和续写结果都是真实生成结果；“试听”只播放已保存音频，不额外请求供应商生成试听版，也不新增一次音乐生成任务。
- 每首音频记录音频地址、下载地址、格式、时长和供应商结果编号。
- 每首音频都有独立下载操作；优先让用户通过浏览器保存文件选择器指定位置，不支持时使用浏览器默认下载。
- 正式音乐供应商若无法提供稳定可访问的试听或下载资源，则不满足首期验收要求。

## 2026-07-15：先试听首段再决定续写

结论：P7 支持“生成首段初稿 -> 进入试听区 -> 用户试听 -> 主动继续续写”的两阶段流程，但不把初稿伪装成供应商提供的特殊试听版。

- 试听不会调用 AI；点击“继续续写”才会创建一次新的外部生成任务并产生新的费用和状态。
- 续写任务必须引用来源结果，保存供应商歌曲编号、续写位置和新增歌词，且不能覆盖原始初稿。
- 原始初稿和续写结果均独立保留、播放和下载。
- Mureka 当前官方 API 明确提供歌曲续写能力，优先进入 P7 实测；是否最终选用仍取决于中文歌曲质量、续写连贯性、价格、稳定性和商用条款。
- 统一 `MusicProvider` 只把续写定义为可选能力；不具备该能力的供应商仍可用于一次性完整歌曲生成。

## 2026-07-15：统一 API 用量账本与余额看板

结论：所有外部 AI 能力共用 `api_usage_records` 调用账本，任务业务表只保存业务状态，不为每个供应商重复增加 Token 字段。

- 每次调用记录任务、操作、供应商、模型、请求方法、接口、供应商请求编号、耗时、重试次数、状态和原始用量摘要。
- 文本模型使用供应商响应中的输入、输出、缓存和总 Token；缺失时保持未知或 0，不使用字符数推算并冒充官方数据。
- 音乐平台按官方返回的积分、生成次数或音频时长记录，Token 只用于真正以 Token 计量的接口。
- Dashboard 展示今日、近 7 日、按供应商汇总和最近调用；旧任务不回填虚构数据。
- 每日用量按日期倒序展示，默认只显示最近 3 天；更早的近 7 日记录保留在页面中，由用户按需展开。
- 自动余额只允许通过供应商官方接口获取。Mureka 文档提供 `/v1/account/billing`；当前智谱兼容接口只接入调用响应用量，余额暂由超级管理员前往控制台查看。
- 余额查询和业务生成解耦，查询失败不能阻断分析、作词或音乐生成任务。

## 2026-07-15：长列表默认收起规则

结论：运行记录、任务记录和接口调用等历史列表不能把全部数据常驻铺开，统一使用“最新记录预览 -> 按需展开 -> 再次收起”的交互。

- 页面级历史记录默认显示最新 5 条，其余记录通过列表底部按钮展开。
- Dashboard 的最近接口调用先按任务类型分为榜单分析、歌词创作、音乐创作和接口测试等折叠组；每次只展开一个组，组内不混入其他任务类型。
- 单个任务内的接口调用明细默认显示最新 3 条，同时保持完整调用顺序可追溯。
- 榜单歌曲、分析选歌等需要连续浏览的数据继续使用分页；榜单快照继续使用单块折叠，不重复套用第二层收起控件。
- 账号等管理对象使用分页，不因默认收起导致用户误以为数据不存在。
- 展开和收起按钮在桌面与手机端使用统一样式，并明确显示尚未展示的记录数量。

## 2026-07-15：文本模型配置热切换

结论：分析和作词不再只依赖进程启动时读取的单组 `AI_*` 环境变量，改为“代码模板 + 数据库连接配置 + 环境变量后备”三层结构。

- 预置本地规则、智谱 BigModel、DeepSeek、阿里百炼 Qwen、MiniMax 和通用 OpenAI-compatible 模板；模型名允许按账号实际能力修改。
- API Key 使用后端主密钥认证加密，数据库只保存密文和末尾提示，任何查询接口都不返回明文。
- 配置必须先完成低 Token 连接测试，测试成功后才能设为唯一活动项；活动项不能直接编辑或删除。
- 分析和作词在每次调用开始时解析活动配置，随后持有该 Provider 完成整次调用。运行中切换只影响后续调用，不重启服务，也不改变已经开始的任务。
- `.env` 保留为首次部署和数据库无活动配置时的后备，可由超级管理员一次性导入；自动化测试继续默认使用本地规则。
- 连接测试本身可能产生费用，因此同样写入 `api_usage_records` 并展示真实 Token。

## 2026-07-15：团队收藏夹与具体产物版本

结论：收藏夹是内部团队共享的精选产物区，首期支持榜单分析报告和歌词版本两类，后续音乐结果沿用同一分类结构扩展。

- 分析收藏指向具体 `analysis_reports.id`，歌词收藏指向具体 `lyrics_versions.id`，不收藏会继续变化的任务外壳。
- 重新生成歌词不会覆盖或替换已经收藏的旧版本。
- 收藏只保存目标类型、目标编号、收藏人、收藏时间和备注；报告与歌词正文继续以原业务表为唯一数据源。
- 每种产物全团队只保留一条收藏关系，重复点击按幂等处理。
- 收藏页展示来源任务、具体版本、供应商、模型、任务 Token、产物生成时间、收藏人和收藏时间，并按类别筛选、按 8 条分页。
- 备注最多 2000 字，默认隐藏，同一时间只展开当前编辑项；移出收藏夹不会删除原始分析或歌词。
- 被收藏产物视为主动保留内容；后续实现 30 天普通产物清理时必须跳过收藏目标，移出收藏后才重新进入普通保留规则。
- 作词任务记录默认显示最新 3 条，保证当前已有少量数据时也能直接看到展开和收起入口。

## 2026-07-15：线性可配置自动工作流

结论：在固定标准流程基础上增加轻量工作流编排。用户可以从当前已实现的采集、分析、作词 Agent 中选择步骤、保存流程模板并一键运行；这次范围调整覆盖原先“首期不做自定义工作链”的旧决定。

- 当前只支持符合数据依赖的线性顺序：采集 -> 分析 -> 作词；用户可以省略不需要的前置步骤，但自动作词必须包含分析。
- 工作流不是只按顺序触发按钮。采集步骤的 `snapshot_id` 用于取出本次歌曲 `entry_ids` 并传给分析；分析步骤的 `report_id` 作为作词的 `analysis_report_id`；作词产出的 `lyrics_version_id` 为 P7 音乐生成预留。
- 流程模板、流程运行和每个步骤运行分别持久化。步骤记录 `task_id`、`output_id`、状态、时间和错误，便于沿数据链定位问题。
- 后台任务先返回运行编号，前端每 2.5 秒轮询；同一时间只允许一条自动流程，避免免费文本接口并发触发限流。
- 分析完成后默认等待 15 秒再开始作词，等待时间由环境变量配置；失败时停在当前步骤，不执行下游，也不产生假成功。
- 步骤权限沿用现有 Agent 权限，模板团队共享；没有对应权限的成员不能创建或运行包含该步骤的流程。
- 音乐创作步骤在正式供应商接入前保持禁用。P7 接入后使用歌词版本的统一创作方案继续传参。
- 当前不实现拖拽画布、任意排序、条件分支、并行节点、循环、跨类型参数手工映射或总调度 AI，这些仍属于高级增强范围。
