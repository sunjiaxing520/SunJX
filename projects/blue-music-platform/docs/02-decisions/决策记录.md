# 蓝乐 AI 音乐创作平台决策记录

## 2026-06-02：首期爬虫数据源

结论：基础版首期先爬 **酷狗热歌榜**。

原因：

- 报价方案中基础版已明确提到酷狗热歌榜。
- SRS 中曾出现 QQ 音乐描述，后续以本决策记录和报价方案为准。
- 后续如进入完整版，再扩展 QQ 音乐、网易云音乐等多源爬取。

开发影响：

- P1/P2 之后的爬虫 Agent 优先实现酷狗热歌榜数据采集。
- 数据模型仍保留 `platform` 字段，为后续多源扩展预留。
- 文档、任务说明和 Cursor 提示词中统一称为“酷狗热歌榜”。

## 2026-06-02：首期创作工具

状态：已被 2026-07-11“官方音乐生成 API”决策取代。

结论：初步暂定使用 **Suno** 作为首期音乐创作工具。

说明：

- 平台不直接承诺“完全原创替代人工作曲”，而是把用户输入和分析结果整理成更适合 Suno 使用的创作素材。
- 对客户讲解时尽量使用中文字段，不直接讲英文接口参数。
- 后续如果 Suno 接口或平台规则变化，应保留创作工具适配层，避免业务逻辑和单一平台强绑定。

首期创作需要准备的中文素材：

```text
歌曲标题
创作类型：带人声 / 纯音乐
歌词：用户填写或 AI 辅助生成
音乐风格：流行、摇滚、电子、民谣、R&B 等
情绪氛围：欢快、悲伤、舒缓、激昂等
节奏速度：慢速 / 中速 / 快速，或 BPM 范围
人声要求：男声 / 女声 / 不指定
补充要求：用户自由描述
```

完整版可扩展：

```text
歌曲结构：主歌 / 副歌 / 过渡段
不想要的风格
多个创作版本
在线试听
版本回退
```

## 2026-06-02：Agent 操作模式

状态：已被 2026-07-11“首期自动工作流和 Agent 分工”决策取代。

结论：首期先做 **人操作多个 Agent**，后续再升级为 **总调度 Agent 调用多个专业 Agent**。

首期模式：

```text
用户分别操作爬虫 Agent、分析 Agent、创作 Agent。
```

后续模式：

```text
用户只提出目标，总调度 Agent 自动拆任务并调用多个专业 Agent。
```

原因：

- 首期先保证客户能看懂、能操作、能验收。
- 人工操作模式开发范围更清晰，风险更低。
- 总调度 Agent 可以作为后续增强能力，不需要推翻现有多个专业 Agent 的设计。

边界说明：

- 首期 Agent 更偏“自动化任务机器人”，例如采集榜单、提交表单、跟踪任务、保存结果。
- 后续增强 Agent 才考虑引入更强的 AI 推理、长期记忆和自动调度能力。
- 自定义工作链、Agent 记忆系统、总调度 Agent 均放入初步设想，不纳入当前首期交付范围。

## 2026-06-02：新增作词 Agent

状态：保留作词 Agent，但基础记忆和多个作词 Agent 调整为后续增强范围。

结论：首期增加 **作词 Agent**。

原因：

- 客户希望用 AI 写歌词。
- Suno 创作带人声歌曲时，歌词是核心素材。
- 作词质量和客户偏好强相关，所以作词 Agent 需要基础记忆。

首期作词 Agent 的能力：

```text
根据主题生成歌词
根据风格生成歌词
根据情绪生成歌词
支持重新生成
支持局部修改副歌/主歌
支持把歌词送入 Suno 创作 Agent
```

首期基础记忆：

```text
用户常用主题
用户偏好的歌词风格
常用表达方式
不想要的词或风格
历史歌词记录
```

边界：

- 首期只做基础偏好记忆，不做复杂长期记忆系统。
- 多个作词 Agent、不同作词 Agent 拥有不同记忆，放入后续增强项目。
- 总调度 Agent 自动选择作词 Agent，也放入后续增强项目。

## 2026-07-11：首期自动工作流和 Agent 分工

结论：首期采用 **一条固定自动工作流**，用户点击一次后，系统按固定顺序自动执行各个 Agent。

```text
酷狗榜单采集
-> 榜单趋势分析
-> AI 作词
-> 官方音乐生成 API
-> 保存结果并展示
```

实现边界：

- 工作流调度器统一控制执行顺序，Agent 之间不直接互相调用。
- 爬虫、分析、作词、音乐生成仍保持独立模块，分别定义清楚输入和输出。
- 首期每种 Agent 只有一个默认实现，不做用户选择多个 Agent。
- 首期不做总调度 AI、自定义工作链、Agent 热插拔和复杂长期记忆。
- 每一步保存状态和结果，失败时记录原因，并允许重新执行。
- 前端使用轮询查看任务状态，首期不增加 WebSocket 和复杂实时系统。

后续扩展时，可以在同一工作流中为不同步骤选择不同的 Agent 实例，但不影响首期固定流程。

## 2026-07-11：内部团队、账号和数据范围

结论：首期是 **单客户内部团队工具**，不是面向公众开放注册的 SaaS。

- 系统不开放用户自助注册。
- 初始化一个超级管理员，由超级管理员创建、停用账号并分配 Agent 使用权限。
- 榜单、分析、歌词、任务和创作结果默认在团队内共享。
- 每条任务和结果仍记录创建人，便于追踪来源。
- 榜单按自然日保存快照，默认保留最近 30 天，页面最多展示最近 15 次运行记录。
- 同一天重复采集时更新当天快照，不伪造额外历史天数。
- 榜单分析默认使用连续 7 天快照；实际不足 7 天时按现有天数分析并降低证据可信度。
- 普通任务产物默认保留 30 天，后续实现时再确定主动保存作品是否长期保留。

## 2026-07-11：官方音乐生成 API

结论：首期不使用非官方 Suno 套壳接口或网页自动化，统一通过官方音乐生成 API 适配层接入。

当前候选顺序：

1. Mureka 官方 API：首选候选，支持中文歌词、完整歌曲、异步任务状态和商业授权。
2. MiniMax Music 官方 API：备选候选，支持中文歌词和完整歌曲生成。

开发要求：

- 在客户提供正式账号前，先用开发测试账号完成同一批中文歌曲对比测试。
- 根据中文咬字、音乐质量、生成时间、水印、价格和稳定性确定最终首期供应商。
- 业务模块只依赖统一的 `MusicProvider` 接口，不在核心流程中写死供应商名称。
- 首期只真正接入一个供应商，不做多供应商自动切换。
- API Key 只保存在后端环境变量中，不进入前端和 Git 仓库。

## 2026-07-13：分析和作词按音乐平台输入反向设计

结论：P5 和 P6 不只输出说明文章，而是共同生成可供正式音乐生成 API 使用的结构化创作方案。

- P5 输出曲风、情绪、主题、场景、速度档位、人声方向、乐器方向、结构和 Hook 建议。
- P6 输出标题、标准段落标签歌词和风格提示词。
- 不从榜单元数据假装推断准确 BPM、调性、和弦或真实编曲。
- P7 只负责把统一创作方案映射成最终供应商参数，避免业务流程绑定单一平台。
- 当前开发环境使用明确标记的本地规则基线；正式文本模型通过 OpenAI-compatible 适配器和后端环境变量接入。
